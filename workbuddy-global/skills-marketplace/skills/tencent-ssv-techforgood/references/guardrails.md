# 自检护栏

> **更新日期**：2026-04-14
> **用途**：每个 Step 输出前做最小必要自检。

## 一、Step 级自检

### Step 1

1. 是否使用了正确的渠道交互方式？（IDE 用交互工具；IM 用编号文本）
2. 路径 B 是否只采集 `org_type`、`org_field`、`org_size`？
3. 路径 A 预填是否坚持“官网优先，其他来源待确认”？
4. 是否避免在 Step 1 提前推荐具体产品？
5. 是否没有提前读取产品目录快照？
6. 做完 Step 1 后，本轮是否已收口等待用户回复？

### Step 2

1. 是否使用了正确的渠道交互方式？
2. 是否只采集 `pain_points` 和 `urgency`？
3. 是否先复述了 Step 1 的机构信息？

### Step 3

1. 是否只用两个固定 URL 获取产品数据？
2. 是否判断了“抓取失败”与“结果明显不完整”两种情况？
3. 满足以下任一条件时，是否读取了 `references/digital_toolbox.md` 兜底？
   - 任一 URL 抓取失败
   - 腾讯自有产品少于 **8** 个，或缺少 `腾讯文档` / `企业微信` / `腾讯会议` / `腾讯问卷` 中任一关键产品
   - 第三方产品少于 **5** 个，或缺少 `益帮帮-项目执行管理系统` / `金数据公益企业版` / `音书-手语翻译服务基础版` 中两项及以上
   - 页面只有导航、营销文案或零散片段，无法形成稳定产品清单
4. 兜底时是否保留了在线结果中的实时字段，只补本地缺失字段？

### Step 4

1. 推荐产品是否全部来自 Step 3 的工具箱结果？
2. 是否只推荐了 3-8 款产品？
3. 是否遵循“领域相关 → 免费腾讯产品优先 → 第三方产品”的排序？

### Step 5

1. 每张详情卡是否包含：推荐理由 / 产品详情 / 申领入口？
2. 教程、案例、实施参考是否先经 `references/knowledge_index.md` 路由，并按当前匹配方案从官网 Agent 训练知识库总表或已命中的在线文档**实时**生成，而不是引用本地静态教程库？
3. 在 WorkBuddy 环境下，若宿主已安装或可启用 `playwright-cli` / Playwright 套件，命中在线文档后是否先直用或启用模拟浏览器，再等待页面渲染后判断正文是否稳定可读？
4. “正文稳定可读”的判断是否足够严格：不是只有标题、导航、登录提示或壳页，而是已读到与目标产品或主题直接相关的正文段落、步骤、表格或说明？
5. 若浏览器能力最终不可用，或总表、索引表、正文只命中壳页，是否已透明降级为产品详情页、官方帮助中心或 `references/knowledge_index_by_category.md` 中的官方文档链接 + 适用说明？
6. 申领地址是否只核对 `references/digital_toolbox.md`？
7. 若当前只能确认“登录后在详情页点击立即申领”，是否如实说明，没有编造隐藏直链？
8. 链接格式是否符合渠道要求？（IDE 超链接；IM 纯文本 URL）

### Step 6

1. 是否输出了清单速览？
2. 清单中的每个产品是否都有申领入口？
3. 是否附上了申领须知？
4. 是否发起了后续操作交互？

## 二、全局自检

1. **渠道一致性**：IDE 全程使用交互工具 + Markdown 超链接 / 表格；IM 全程使用编号文本 + 纯文本 URL + 缩进列表
2. **主线一致性**：只围绕公益机构数字化赋能，不扩展成长篇 FAQ
3. **收尾完整性**：推荐过任何工具就必须有清单速览
4. **敏感信息提醒**：涉及身份证号、手机号、签字盖章件、AppSecret、AppCode 等内容时，必须附安全提醒

## 三、法规引用自检

1. 必须先查权威来源，不默认依赖任何本地静态法规快照
2. 引用法规时必须附免责声明：`📌 以上法律法规信息仅供参考，不构成法律意见。请以 flk.npc.gov.cn 最新版本为准。`
3. 涉及比例、门槛、时间节点时，必须给出具体数字和条文依据
4. 若当前无法实时核验法规原文，应明确提示用户以国家法律法规数据库及主管部门最新发布为准，不要凭记忆输出关键结论
